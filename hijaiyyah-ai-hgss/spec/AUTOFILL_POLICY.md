# AUTOFILL POLICY — Hijaiyyah-AI HGSS VM v1.0

Status: NORMATIVE / EVALUATION POLICY

## 1. Objective
This document defines the scope and legal/technical nature of the artifacts generated by the `Guarded Mode (Mode B)` deterministic autofill mechanism in Hijaiyyah-AI HGSS VM v1.0.

## 2. Non-Production Disclaimer
> [!IMPORTANT]
> All binary artifacts generated by this version (v1.0) are **deterministic synthetic artifacts** designed for compliance evaluation (Industrial Harness). 

They are **NOT** live financial transactions, production cryptographic proofs, or legally binding signatures. They serve as a "Compliance Oracle" to test the AI's ability to generate semantically correct metadata.

## 3. Artifact Nature
The following files generated in `artifacts/runs/` are synthetic:
- `ct.bin`: Deterministic bytes seeded by `txid_hex`.
- `trace.jsonl`: Deterministic execution trace.
- `lease_token.bin`: Static demo token from `deps/hgss-hc18dc/artifacts/`.

## 4. Hash Determinism
The purpose of `autofill.py` is to allow the LLM to focus on **semantic field correctness** (e.g., `txid_hex`, `timestamp_utc`, `lease_id`) while the system handles the bit-accurate recomputation of digests (`SHA-256`) and canonical serialization (`CBOR`).

## 5. Pass@k Criteria
In this harness, a case is considered `PASS` only if the resulting `event_single.json` (after autofill) is successfully accepted by the normative oracle `hgss_verify_evidence.py` without errors.

## 6. Transition to Production
Future versions (v2+) will replace the deterministic harness with real-time HSM binding and dynamic ciphertext generation. v1.0 is strictly for **Industrial Compliance Benchmarking**.
